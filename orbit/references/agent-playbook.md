# Agent Playbook

`orbit`의 view 에이전트와 Orchestrator가 공통으로 따를 상세 지침이다.
각 view 문서는 이 문서를 전제로 하고, view 특화 조건만 추가한다.

## 공통 원칙

- observation은 사실 관찰만 적는다. 점수나 triage 판단은 적지 않는다.
- claim은 한 문장으로 쓴다. 한 claim에 서로 다른 문제를 두 개 이상 섞지 않는다.
- evidence는 직접 읽은 `file:line`만 적는다.
- `읽지 못함`, `아마`, `추정`, `가능성`만 있는 문장은 claim으로 승격하지 않는다.
- generated 산출물, lockfile, 빌드 결과물은 보조 근거로만 쓰고, 가능하면 source 설정 파일을 같이 적는다.

## 조사 우선순위

1. 진입점과 설정 파일
2. CI, 배포, 상태관리, 라우터 같은 공통 경로
3. 실제 feature 구현 파일
4. 테스트와 문서

우선순위가 높은 파일에서 이미 충분한 반례가 나오면 하위 탐색은 줄여도 된다.

## finding으로 올릴 최소 조건

아래를 모두 만족할 때만 finding 후보로 본다.

- 문제 설명이 한 문장으로 닫힌다.
- 직접 읽은 evidence가 1개 이상 있다.
- 영향 범위를 `impact_surface` 한 문장으로 설명할 수 있다.
- `next_step`에 파일 경로 또는 식별자를 넣을 수 있다.

아래는 finding 후보가 아니다.

- 스타일 취향 차이
- 팀 규칙 문서 없이 일반론만으로 지적한 경우
- 현재 구조의 trade-off로 설명 가능한 경우
- source 근거 없이 generated 결과만 보고 단정한 경우

## 선언 경로 vs 도구 실제 경로 교차 확인

설정 파일(CI, Dockerfile, compose, 런타임 설정)에서 경로를 선언했을 때,
**그 경로를 실제로 쓰는 도구가 같은 경로를 기본값으로 사용하는지 반드시 교차 확인한다.**

이 패턴은 BUILD·OPS·DEP 등 어떤 view에서든, 설정 파일과 실행 커맨드를 같이 볼 때마다 적용한다.

### 언제 확인하나

아래 중 하나라도 해당하면 교차 확인을 수행한다.

- CI 설정에 `cache:`, `volumes:`, `paths:` 또는 환경변수로 경로를 선언한 경우
- Dockerfile에 `WORKDIR`, `COPY`, `RUN --mount=type=cache,target=` 같이 특정 경로를 지정한 경우
- 도구 설정 파일(`.npmrc`, `pnpm-workspace.yaml`, `pyproject.toml` 등)에 저장소·캐시 경로를 설정한 경우

### 확인 방법

1. 설정 파일에서 경로 선언을 찾는다.
2. **같은 파일 또는 연관 파일**에서 해당 도구의 실행 커맨드를 찾는다.
3. 커맨드에 경로를 명시하는 플래그가 없으면, 도구의 **기본 저장 경로**가 선언된 경로와 일치하는지 확인한다.
4. 불일치하면 finding 후보로 올린다.

evidence는 **경로 선언 위치(파일:줄) + 도구 호출 위치(파일:줄)** 두 가지를 반드시 포함한다.

### 대표 사례

| 도구 | 기본 경로 (플래그 없을 때) | 경로 지정 플래그 |
|------|---------------------------|-----------------|
| pnpm | `~/.local/share/pnpm/store` | `--store-dir <path>` |
| npm | `~/.npm` | `--cache <path>` |
| pip | `~/.cache/pip` | `--cache-dir <path>` |
| Gradle | `~/.gradle/caches` | `-g <path>` 또는 `GRADLE_USER_HOME=<path>` |
| Docker buildx | `~/.docker/buildx` | `--cache-from type=local,src=<path>` |
| Go module | `~/go/pkg/mod` | `GOPATH=<path>` 또는 `GOMODCACHE=<path>` |

이 표는 예시일 뿐이며, **표에 없는 도구라도 기본 경로가 선언된 경로와 다를 수 있다.**
모르는 도구가 나오면 해당 도구의 `--help` 또는 공식 문서에서 기본 경로를 확인한 뒤 판단한다.

### finding으로 올릴 조건

- 경로 선언이 있고, 도구가 그 경로를 기본값으로 사용하지 않으며, 실행 커맨드에 경로 플래그가 없다.
- 선언된 경로가 실제 효과 없이 비어 있을 가능성이 있다 (캐시 무효화, 볼륨 미마운트 등).

### 문제로 보지 않는 경우

- 도구의 기본 경로가 선언된 경로와 일치하는 경우.
- 도구가 환경변수로 기본 경로를 오버라이드하고 있고, 해당 환경변수가 선언된 경로를 가리키는 경우.
- `$(pnpm store path)` 같이 경로를 동적으로 읽어 선언에 주입하는 경우.

## 반례 처리

반례가 하나라도 나오면 아래를 먼저 확인한다.

- 반례가 최신 경로인가
- source와 generated 중 어느 쪽이 기준 원천인가
- 테스트, CI, runtime 경로가 같은 사실을 가리키는가

반례가 claim을 완전히 뒤집으면 rebuttal 후보로 올리고, 일부만 약화하면 claim 범위를 좁힌다.

## next_step 품질 규칙

`next_step` 품질 채점 공식과 통과 기준은 **`SKILL.md` Step 5**에 단일 정의되어 있다.
여기서는 감각을 잡기 위한 예시만 제공한다.

나쁜 예 (actionability 점수 낮음):

- "구조를 개선한다" — 파일 경로, 식별자, 명령어 없음
- "리팩터링을 검토한다" — 행동이 모호하고 끝이 없음
- "문서를 보강한다" — 어느 파일의 어떤 내용인지 없음

좋은 예 (actionability 3점 이상):

- "`src/features/auth/model/store.ts`의 `persistSession` 중복 저장을 제거하고 `pnpm test auth`로 회귀를 확인한다."
  → 파일 경로(+2) + 식별자(+1) + CLI(+1) + 한 문장(+1) = 5점

## 축소 조사 규칙

완전 스킵 전에 아래를 먼저 검토한다.

- 주된 조사 대상이 없어도 인접 설정 파일로 축소 조사 가능한가
- 같은 문제를 다른 레이어에서 확인할 수 있는가
- 정적 파일만 있어도 문서나 build 설정으로 간접 검증 가능한가

축소 조사 후에도 근거가 없으면 그때 스킵한다.

## 제외해도 되는 것

- vendor 코드
- coverage 산출물
- `.next`, `dist`, `build` 같은 생성 산출물
- 이미지, 폰트 같은 binary asset

단, 문서 drift나 빌드 drift를 판단할 때는 산출물 경로를 보조 근거로 적을 수 있다.

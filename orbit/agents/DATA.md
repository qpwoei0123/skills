# DATA

데이터 구조 & 흐름 view다.
데이터가 단일 출처에서 예측 가능한 경로로 흐르는지 본다.

## Agent A — 스토어 중복 상태 + 비정규화 저장

찾을 것:

- 동일 엔티티가 여러 슬라이스에 중복 저장된 경우
- 서버 응답을 변환 없이 그대로 스토어에 적재하는 패턴
- 화면별로 같은 데이터를 다른 키로 저장하는 경우

## Agent B — 파생 상태 남용 탐지

찾을 것:

- selector나 computed로 만들 값을 영속 상태로 저장하는 경우
- 동기화가 필요한 상태가 2곳 이상에 존재하는 경우
- 캐시와 UI 상태가 섞여 있는 구조

## Agent C — API 응답 → 스토어 변환 레이어

찾을 것:

- 응답을 스토어에 넣기 전 변환 레이어 존재 여부
- 페이지마다 엔티티 형태가 달라지는 경우
- DTO와 도메인 모델 경계가 없는 경우

## 우선 조사 경로

- 상태관리 파일, query hook, API client, selector
- 동일 엔티티 이름이 반복되는 slice와 화면
- 응답 타입 정의와 UI가 실제 소비하는 필드

## finding으로 올릴 최소 조건

- 같은 데이터가 두 군데 이상에서 다른 구조로 저장되거나 계산된 값을 영속 저장한다.
- 응답 원본과 화면 소비 형태가 뒤섞였다는 근거를 적을 수 있다.
- 정리 대상 store, selector, adapter 이름을 `next_step`에 넣을 수 있다.

## 문제로 보지 않는 경우

- 페이지 로컬 state 하나에만 있는 파생 값
- 성능상 의도된 캐시 키 분리
- API 응답 타입과 도메인 타입이 같아 별도 변환이 필요 없는 경우

## 백엔드 레포 — 데이터 흐름 대체 분석

상태관리 라이브러리가 없는 백엔드·서버사이드 레포에서는 아래로 대체한다.

### Agent A 대체 (DB 쿼리 패턴)

찾을 것:
- 루프 안에서 개별 쿼리를 실행하는 N+1 패턴
- 단일 레코드 조회가 반복되는 경우 (배치 조회로 대체 가능한 경우)
- ORM/쿼리빌더 결과를 중간 변환 없이 바로 응답으로 내보내는 경우

### Agent B 대체 (검증 레이어 일관성)

찾을 것:
- 외부 입력(HTTP 요청 body/query/param)을 검증 없이 직접 사용하는 경우
- DTO나 스키마 검증이 일부 라우트에만 적용된 경우 (불일치)
- DB 스키마와 애플리케이션 모델 타입이 다른 구조를 가지는 경우

### Agent C 대체 (응답 형태 일관성)

찾을 것:
- 같은 엔티티가 엔드포인트마다 다른 필드 구조로 응답되는 경우
- 에러 응답 형태가 라우트마다 다른 경우
- 직렬화(serialization) 레이어 없이 ORM 객체를 그대로 반환하는 경우

## 축소 조사 규칙

- 상태관리 라이브러리가 없는 **프론트엔드**라면 query cache와 컴포넌트 state를 중심으로 본다.
- **백엔드** 레포라면 위 "백엔드 레포" 분석으로 A·B·C를 대체한다.
- 정적 페이지처럼 보여도 form state나 search param 동기화가 있으면 일부 조사한다.

## 스킵 조건

- 상태관리 없음 **AND** 백엔드 패턴도 없음 → A·B 스킵, C만 (응답 일관성)
- 단순 정적 문서 사이트, HTML 파일만 있는 경우 → 전체 스킵

## Orchestrator 메모

- 중복 상태와 파생 상태 남용은 한 finding으로 병합될 가능성이 높다.
- 변환 레이어 부재는 시스템 전체 데이터 일관성 문제로 볼 수 있다.
- 백엔드에서 N+1과 검증 누락이 함께 나오면 데이터 신뢰성 리스크로 묶을 수 있다.
- "상태관리 없음"이라고 해서 바로 스킵하지 말고, 레포 성격을 먼저 파악한다.

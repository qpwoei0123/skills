# OPS

운영 관측성 view다.
장애 시 로그·알림·복구 경로를 빠르게 찾을 수 있는지 본다.

## Agent A — 로그 출력 위치 + 레벨 일관성

찾을 것:

- `console.log`, `warn`, `error` 사용 패턴
- 로깅 유틸리티 통일 여부
- 민감 정보가 로그에 직접 찍히는지

## Agent B — 알림/모니터링 설정 파일

찾을 것:

- Sentry, Datadog 같은 에러 트래킹 설정 존재 여부
- 알림 임계값 또는 샘플링 설정
- 운영용 DSN, key, env 분리 상태

## Agent C — 장애 복구 경로

찾을 것:

- rollback 스크립트 또는 절차 문서 존재 여부
- fallback UI 또는 error boundary 존재 여부
- 장애 시 사용자 영향 완화 장치

## 우선 조사 경로

- 로깅 유틸, 에러 처리 공통 모듈
- Sentry, Datadog, OpenTelemetry 같은 추적 설정 파일
- 에러 화면, fallback UI, runbook 문서

## finding으로 올릴 최소 조건

- 로그, 모니터링, 복구 경로 중 하나가 핵심 흐름에 비해 비어 있거나 불일치한다.
- 장애 시 누가 무엇을 볼 수 없는지 한 문장으로 설명할 수 있다.
- 수정 대상 설정 파일, boundary 컴포넌트, runbook 경로를 `next_step`에 적을 수 있다.

## 문제로 보지 않는 경우

- 로컬 개발용 `console.log` 몇 줄만으로 운영 리스크를 과장하는 경우
- 외부 SaaS 추적 설정이 비공개 env라서 코드에 직접 없다고 부재로 단정하는 경우
- 프론트엔드 전용 프로젝트에 rollback script가 없다는 이유만으로 즉시 finding 처리하는 경우

## 축소 조사 규칙

- 모니터링 설정 파일이 없어도 공통 에러 핸들러와 사용자 fallback UI를 먼저 본다.
- 백엔드가 없는 저장소면 복구 절차 대신 사용자 영향 완화 장치를 중심으로 본다.

## 스킵 조건

- 프론트엔드 전용 → C를 error boundary 탐지로 대체
- 로그 파일 없음 → A만 깊게

## Orchestrator 메모

- 모니터링 부재와 복구 경로 부재가 함께 있으면 운영 리스크를 높게 본다.
- 프론트엔드 전용 프로젝트라도 장애 완화 장치는 찾는다.

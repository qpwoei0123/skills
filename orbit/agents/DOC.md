# DOC

지식 내구성 view다.
문서와 결정 기록이 팀 밖으로 사라지지 않는지 본다.

## Agent A — README + 온보딩 문서 최신 여부

찾을 것:

- README 존재 여부와 내용 최신성
- 실행 방법, 개발 환경, 배포 흐름 설명 여부
- 온보딩 문서 존재 여부

## Agent B — 의사결정 기록

찾을 것:

- ADR, CHANGELOG, 주요 PR 설명 존재 여부
- 아키텍처 결정 근거가 코드 밖에 남아 있는지
- 릴리스 기록과 실제 코드 경로 연결 여부

## Agent C — 복잡한 로직 인라인 주석 밀도

찾을 것:

- 100줄 이상 파일의 주석 밀도
- 비즈니스 규칙 설명 주석 존재 여부
- 이름만으로 의미가 안 드러나는 알고리즘 설명

## 우선 조사 경로

- 루트 README, 온보딩 문서, ADR, CHANGELOG
- 배포나 실행 방법이 적힌 문서
- 길고 복잡한 핵심 로직 파일

## finding으로 올릴 최소 조건

- 문서 부재 또는 노후화가 실제 실행/이해 실패로 이어지는 근거가 있다.
- 코드 밖에 남아야 할 결정 근거가 전혀 보이지 않는다고 설명할 수 있다.
- 보강해야 할 문서 경로나 주석이 필요한 코드 위치를 `next_step`에 적을 수 있다.

## 문제로 보지 않는 경우

- 문서가 짧아도 현재 구조와 실행 방법이 정확한 경우
- self-explanatory한 단순 로직에 주석이 없는 경우
- ADR이 없어도 PR 설명과 CHANGELOG로 결정 흔적이 충분한 경우

## 축소 조사 규칙

- 문서 파일이 적어도 README와 최근 변경 이력은 먼저 본다.
- 주석 밀도는 파일 길이보다 규칙 복잡도 기준으로 판단한다.

## 스킵 조건

- 문서 파일 0개 → B·C 스킵, A만 깊게

## Orchestrator 메모

- README 부재와 결정 기록 부재가 함께 나오면 온보딩 리스크로 묶을 수 있다.
- 주석 밀도는 스타일 취향이 아니라 규칙 설명 부재 중심으로 본다.

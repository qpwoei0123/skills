# SAFE

변경 안전성 view다.
핵심 흐름에 테스트와 검증 게이트가 있는지 본다.

## Agent A — 테스트 파일 탐색 + 커버리지 분석

찾을 것:

- `*.test.*`, `*.spec.*` 파일 목록
- 커버리지 설정 파일과 threshold
- 핵심 경로에 대응하는 테스트 존재 여부

관찰 예:

- 핵심 경로 컴포넌트는 있는데 대응 테스트가 없음
- 커버리지 설정은 있으나 threshold가 비어 있음

## Agent B — CI 파이프라인 게이트 확인

찾을 것:

- merge 전 필수 통과 스텝 존재 여부
- 테스트·린트·빌드 자동 실행 여부
- pull request 단위 검증 경로 존재 여부

관찰 예:

- CI는 있으나 테스트 잡이 빠져 있음
- 로컬 스크립트와 PR 검증 경로가 다름

## Agent C — 핵심 비즈니스 경로 식별

찾을 것:

- 라우터 설정 또는 주요 페이지 진입점
- 인증, 결제, 저장 같은 핵심 흐름
- 테스트 없는 핵심 경로 목록

관찰 예:

- 핵심 진입점이 테스트 없이 직접 배포 경로에 연결됨

## 우선 조사 경로

- 라우터, 엔트리, 저장/제출 흐름 파일
- `package.json`, CI workflow, coverage 설정 파일
- 핵심 흐름과 직접 연결된 테스트 파일

## finding으로 올릴 최소 조건

- 핵심 경로 또는 공통 검증 경로와 직접 연결된다.
- 테스트 부재, threshold 부재, CI 게이트 누락 중 하나가 source 근거로 확인된다.
- "어떤 흐름이 보호되지 않는지"를 한 문장으로 설명할 수 있다.

## 문제로 보지 않는 경우

- 보조 페이지나 실험용 화면만 테스트가 없는 경우
- E2E가 없어도 unit/integration으로 동일 흐름을 충분히 막고 있는 경우
- 로컬 스크립트와 CI 명령이 달라도 결과적으로 같은 검증 집합을 실행하는 경우

## 축소 조사 규칙

- 테스트 파일이 0개여도 B·C를 바로 버리지 말고 CI와 라우터에서 핵심 경로만 확인한다.
- CI가 없으면 pre-push, pre-commit, release script 같은 대체 게이트를 찾는다.

## 스킵 조건

- 테스트 파일 0개 → B·C 스킵, A만 깊게
- CI 없음 → B 스킵

## Orchestrator 메모

- 핵심 경로와 테스트 부재가 함께 잡히면 병합 후보로 본다.
- 테스트가 아예 없으면 "검증 게이트 부재"를 더 크게 본다.

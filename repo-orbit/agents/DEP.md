# DEP

의존성/설정 안정성 view다.
버전과 환경설정이 분산되거나 우연에 기대지 않는지 본다.

## Agent A — 패키지 버전 고정 + 락파일 상태

찾을 것:

- `package.json`의 캐럿(`^`)·틸드(`~`) 범위 사용 여부
- lockfile 존재 여부와 최신성
- 빌드 핵심 패키지의 major drift

## Agent B — 환경변수 선언 위치 분산 여부

찾을 것:

- `.env`, config 파일, 코드 하드코딩 분포
- 동일 변수가 여러 파일에서 중복 선언된 경우
- 환경 이름이 분기마다 다르게 쓰이는 경우

## Agent C — 취약 버전 패턴 탐지

찾을 것:

- 오래된 major 버전 고착
- 알려진 취약 범위에 가까운 패키지
- 런타임/빌드 체인 핵심 의존성의 방치

## 우선 조사 경로

- `package.json`, lockfile, toolchain 설정 파일
- `.env*`, config 파일, 배포 설정 파일
- 런타임 진입점에서 직접 읽는 환경변수 코드

## finding으로 올릴 최소 조건

- 버전 범위, 중복 선언, 환경 이름 drift 중 하나를 직접 읽은 근거로 적을 수 있다.
- 재현성이나 운영 오동작 위험을 한 문장으로 설명할 수 있다.
- 수정 대상 파일 또는 변수 이름을 `next_step`에 넣을 수 있다.

## 문제로 보지 않는 경우

- peer dependency 범위만 보고 실제 drift를 단정하는 경우
- 예제 `.env.example`과 실제 `.env`의 역할 차이를 중복 선언으로 오인한 경우
- 취약 버전 데이터베이스 없이 "오래돼 보인다"만으로 문제 삼는 경우

## 축소 조사 규칙

- lockfile이 없으면 설치 명령, CI 캐시 키, 패키지 매니저 선언을 함께 본다.
- 환경변수 파일이 없으면 코드와 배포 설정에서만 선언 위치를 추적한다.

## 스킵 조건

- lockfile 없음 → A에 병합 처리

## Orchestrator 메모

- 환경변수 분산과 버전 고착이 함께 있으면 재현성 리스크로 묶을 수 있다.
- 취약 버전 자체를 단정하지 말고 직접 읽은 근거 중심으로 적는다.

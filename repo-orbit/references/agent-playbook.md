# Agent Playbook

`repo-orbit`의 view 에이전트와 Orchestrator가 공통으로 따를 상세 지침이다.
각 view 문서는 이 문서를 전제로 하고, view 특화 조건만 추가한다.

## 공통 원칙

- observation은 사실 관찰만 적는다. 점수나 triage 판단은 적지 않는다.
- claim은 한 문장으로 쓴다. 한 claim에 서로 다른 문제를 두 개 이상 섞지 않는다.
- evidence는 직접 읽은 `file:line`만 적는다.
- `읽지 못함`, `아마`, `추정`, `가능성`만 있는 문장은 claim으로 승격하지 않는다.
- generated 산출물, lockfile, 빌드 결과물은 보조 근거로만 쓰고, 가능하면 source 설정 파일을 같이 적는다.

## 조사 우선순위

1. 진입점과 설정 파일
2. CI, 배포, 상태관리, 라우터 같은 공통 경로
3. 실제 feature 구현 파일
4. 테스트와 문서

우선순위가 높은 파일에서 이미 충분한 반례가 나오면 하위 탐색은 줄여도 된다.

## finding으로 올릴 최소 조건

아래를 모두 만족할 때만 finding 후보로 본다.

- 문제 설명이 한 문장으로 닫힌다.
- 직접 읽은 evidence가 1개 이상 있다.
- 영향 범위를 `impact_surface` 한 문장으로 설명할 수 있다.
- `next_step`에 파일 경로 또는 식별자를 넣을 수 있다.

아래는 finding 후보가 아니다.

- 스타일 취향 차이
- 팀 규칙 문서 없이 일반론만으로 지적한 경우
- 현재 구조의 trade-off로 설명 가능한 경우
- source 근거 없이 generated 결과만 보고 단정한 경우

## 반례 처리

반례가 하나라도 나오면 아래를 먼저 확인한다.

- 반례가 최신 경로인가
- source와 generated 중 어느 쪽이 기준 원천인가
- 테스트, CI, runtime 경로가 같은 사실을 가리키는가

반례가 claim을 완전히 뒤집으면 rebuttal 후보로 올리고, 일부만 약화하면 claim 범위를 좁힌다.

## next_step 품질 규칙

`next_step` 품질 채점 공식과 통과 기준은 **`SKILL.md` Step 5**에 단일 정의되어 있다.
여기서는 감각을 잡기 위한 예시만 제공한다.

나쁜 예 (actionability 점수 낮음):

- "구조를 개선한다" — 파일 경로, 식별자, 명령어 없음
- "리팩터링을 검토한다" — 행동이 모호하고 끝이 없음
- "문서를 보강한다" — 어느 파일의 어떤 내용인지 없음

좋은 예 (actionability 3점 이상):

- "`src/features/auth/model/store.ts`의 `persistSession` 중복 저장을 제거하고 `pnpm test auth`로 회귀를 확인한다."
  → 파일 경로(+2) + 식별자(+1) + CLI(+1) + 한 문장(+1) = 5점

## 축소 조사 규칙

완전 스킵 전에 아래를 먼저 검토한다.

- 주된 조사 대상이 없어도 인접 설정 파일로 축소 조사 가능한가
- 같은 문제를 다른 레이어에서 확인할 수 있는가
- 정적 파일만 있어도 문서나 build 설정으로 간접 검증 가능한가

축소 조사 후에도 근거가 없으면 그때 스킵한다.

## 제외해도 되는 것

- vendor 코드
- coverage 산출물
- `.next`, `dist`, `build` 같은 생성 산출물
- 이미지, 폰트 같은 binary asset

단, 문서 drift나 빌드 drift를 판단할 때는 산출물 경로를 보조 근거로 적을 수 있다.
